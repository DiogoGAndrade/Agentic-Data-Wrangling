# Auditable LLM-Powered Assistant for Data Wrangling

A mixed-initiative data preparation system built for a Master's thesis in Data Science
and Advanced Analytics (NOVA IMS). A local large language model proposes a model-aware
cleaning plan for a tabular dataset; a deterministic guardrail layer rewrites or removes
unsafe actions; an execution engine applies the corrected plan and logs full provenance.
The guiding principle: the LLM proposes, the guardrails dispose.

The empirical study behind this code compares five experimental conditions (raw data,
generic cleaning, LLM with dataset context, LLM with model-aware context, and the full
guardrail-enforced system) across nine tabular datasets, eight open-weight LLMs (3B to
14B parameters), and four downstream ML algorithms under five-fold cross-validation.
The result: once guardrails are enforced, downstream performance is invariant to model
size, and the system never regresses below the raw-data baseline across 184 comparisons.

New to this repository and starting from zero? Read [SETUP.md](SETUP.md) first; it walks
through installing every dependency, pulling models, and running both the app and the
evaluation pipeline on a clean machine.

## What it does

- Profiles an uploaded tabular dataset and flags quality issues (missingness, ID-like
  columns, high cardinality, outliers).
- Asks a local LLM (via [Ollama](https://ollama.com)) for a cleaning plan, given the
  dataset profile and the downstream model the data will feed.
- Validates the plan against a typed schema, then rewrites unsafe actions through nine
  guardrail rules (for example, forcing median imputation for KNN, injecting feature
  scaling for linear models, rejecting actions that target columns absent from the
  dataset).
- Executes the corrected plan deterministically and writes a full audit trail: the raw
  LLM output, the sanitised plan, the execution log, and timing.
- Exports the cleaned dataset, the audit log, and the plan for reproducibility.

## Project layout

```
App/            Streamlit user interface (main.py)
engine/         Core logic
  actions.py            Pure cleaning-action executors
  cleaning_pipeline.py  Per-fold pipeline assembly and guardrail enforcement
  prognostics.py        Quality and leakage diagnostics
  provenance.py         Audit-trail logging
  schemas.py             Pydantic schema for the LLM plan
  profile_dataset.py    Dataset profiling
  config.py              Single source of truth for the random seed and dataset list
llm/            LLM interface
  ollama_client.py       Local model calls
  prompt_templates.py    System and plan prompts (C2/C3/C4 conditions)
  json_utils.py          Robust JSON extraction from model output
evaluation/     Experiment harness, condition contexts, and analysis scripts
tests/          Unit tests for the cleaning-action executors
```

`data/`, `debug/`, and `evaluation/outputs/` are intentionally not version-controlled
(see `.gitignore`); they hold raw datasets and the experiment results generated by the
scripts in `evaluation/`. See `evaluation/README.md` for how to regenerate them.

## Quick start

```bash
pip install -r requirements.txt
streamlit run App/main.py
```

Open the local URL Streamlit prints, upload a CSV, choose a model, generate a cleaning
plan, review each step, and execute.

## Tests

```bash
pytest
```

Unit tests for the cleaning-action executors live in `tests/`, covering the common
paths and edge cases the guardrail layer relies on.

## Evaluation conditions

| Condition | LLM | Model-aware | Guardrails | Role |
|-----------|-----|-------------|------------|------|
| C0 | no  | no  | no            | Raw baseline |
| C1 | no  | no  | deterministic | Generic pipeline |
| C2 | yes | no  | no            | LLM, dataset context only |
| C3 | yes | yes | no            | Model-aware LLM, no safety |
| C4 | yes | yes | yes (9 rules) | Proposed system |

The canonical guardrail enforcement path is the inline logic in
`evaluation/prepare_conditions.py`. `evaluation/enforce_c4_v3.py` is an earlier
post-hoc enforcement approach, retained only for provenance of the Phase C runs.

## Reproducing the benchmark

```powershell
powershell -ExecutionPolicy Bypass -File .\run_all_models.ps1
```

This pulls and runs every model in the `$models` list, generates plans for every
condition, runs the downstream cross-validated evaluation, and consolidates results
into `evaluation/outputs/MASTER_RESULTS_TABLE.csv` and `MASTER_PLAN_QUALITY.csv`. See
[SETUP.md](SETUP.md) for the full prerequisites.
